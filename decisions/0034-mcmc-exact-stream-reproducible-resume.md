# ADR-0034: MCMC Exact-Stream Reproducible Resume

**Date**: 2026-07-19
**Status**: Accepted

## Context

`MCMC.state()` (`src/mc/_mcmc.js`) has always returned only `{x, samplingRate, internal}`, where `internal` is produced by each subclass's `_internal()` override. Since ADR-0020, `_internal()` has followed a deliberate "serialize effective state, not adaptation counters" convention (reinforced by ADR-0021, ADR-0029, and an explicit source comment in `hmc.js` calling the dual-averaging accumulator "Ephemeral"): `_internal()` returns the *ready-to-use* adapted quantity (RWM's effective proposal scale, HMC's effective mass matrix), never the raw Robbins-Monro batch counters or covariance/dual-averaging accumulators that produced it. Separately, and never addressed by any prior ADR, the base constructor always builds `this.r = new Xoshiro128p()` fresh and `Math.random()`-seeded — `state()` never captures `this.r`'s stream position, and five of seven subclasses (RWM, AdaptiveMetropolis, HMC, NUTS, MALA) additionally own a second, subclass-private generator `this._q` that is likewise never captured.

The result: reconstructing a sampler from `state()` restores *where the chain is* but not *what it would draw next* — resumed sampling is statistically equivalent but not bit-for-bit identical to an uninterrupted run. Issue #1033 asks for exact-stream reproducible resume, including mid-warm-up checkpoints.

`ran.dist.Distribution.save()`/`.load()` (ADR-0019) already solves the identical problem for distributions: it round-trips `this.r` exactly via `Xoshiro128p.save()`/`.load()`. `Process.path()` independently uses the same `save()`/`.load()` idiom to checkpoint and restore a PRNG around a non-destructive read (`solutions/correctness/2026-07-07-1500-path-prng-save-restore.md`), which established the precedent that *"a `state()` assertion alone cannot prove PRNG purity — it only checks the mathematical state, not the stream position."* `Xoshiro128p.save()`/`.load()` themselves (`src/core/xoshiro.js`) are a complete, exact round-trip of the generator's entire mutable state (a 4-word array, no hidden state) — the primitive this issue needs already exists and is already trusted elsewhere in the codebase.

A design-propose/design-critique loop considered three mechanisms: (1) widen `_internal()` and add a base-class `prng` key, restoring everything through the existing constructor + `initialState.internal` channel; (2) introduce two new protected hooks (`_savePrng()`, `_adaptState()`) to keep `_internal()`'s contract untouched; (3) restore PRNG streams only, deferring adaptation-counter serialization to follow-up issues. The critique found option 2's new hooks added contract surface (doubling `MCMC`'s three-method subclass contract, and awkwardly making the base class delegate its own `this.r` serialization to a subclass-overridable hook) without solving a deeper problem common to all three options, and option 3 explicitly fails the issue's mid-warm-up acceptance criterion. Option 1 was selected.

The critique also surfaced a structural limit common to every option: `RWM._refreshBase()` (`rwm.js`) reads `this.statistics()`, which is derived from the base-class Welford accumulator (`_welford`, in `_mcmc.js`). ADR-0023 pins `_welford`/`_acBuf`/`_acCross`/`_arBuf` as intentionally never serialized, reset on every construction and at the start of `sample()`; changing that "is significant enough to require a superseding ADR" (ADR-0023's own words). No design in this issue's scope can make RWM's step-size adaptation bit-for-bit identical across a mid-batch-window resume without either violating that boundary or refactoring RWM to own a private variance accumulator — both out of scope for #1033. The same accumulators also drive `_thinningLag()`/`_adjustSamplingRate()`, so `samplingRate` (the chain's thinning interval) is likewise not guaranteed to reproduce exactly across a mid-warm-up resume for *any* subclass, independent of RWM.

## Decision

1. **`state()` gains a new top-level `prng` key**: `state().prng = this.r.save()`. The constructor restores it via `if (initialState.prng) this.r.load(initialState.prng)`, applied before `this.x` is derived (so an unprovided `x` is redrawn from the resumed stream, not a fresh one).
2. **Each of the five `_q`-owning subclasses (RWM, AdaptiveMetropolis, HMC, NUTS, MALA) adds a `prngQ` field to `_internal()`**, restored the same way through the existing `initialState.internal` channel. Slice and Gibbs have no `_q` and are unaffected by this point.
3. **`_internal()`'s "effective state only" convention is explicitly reversed**: each subclass's `_internal()` now also serializes its adaptation-batch accumulators (RWM: `_pAccepted`/`_pN`/`_pBatch`; AdaptiveMetropolis: `_covN`/`_covMean`/`_covS`; HMC/NUTS: `_daMu`/`_daHbar`/`_daLogEpsBar`/`_daT` and, for HMC, the mass-matrix accumulator `_metN`/`_metMean`/`_metM2`-or-`_metCovS`; MALA: `_pAccepted`/`_pN`/`_pBatch`; Slice: `_bwSum`/`_adjN`/`_adjBatch`), restored via the same constructor channel with `??`-style fallbacks so snapshots taken before this change keep working. This decision **supersedes** the "serialize effective state, not adaptation counters" guidance in ADR-0020 §2, ADR-0021, and ADR-0029 for the specific fields listed above — those ADRs' broader three-method contract (`_iter`/`_adjust`/`_internal`) and their accumulator-mechanics rationale otherwise stand unmodified.
4. **ADR-0023 is not touched.** The base-class diagnostic accumulators (`_welford`, `_acBuf`/`_acCross`, `_arBuf`/`_arCount`/`_arN`) remain intentionally unserialized. Two consequences are accepted as explicit, documented scope boundaries rather than defects:
   - **RWM's `_base` (per-dimension proposal scale) is not guaranteed bit-for-bit reproducible across a resume that lands mid-batch-window**, because `_refreshBase()` recomputes it from `this.statistics()`, which is derived from the never-serialized Welford accumulator. Every other subclass's adaptation state is fully restored by this ADR. A resume that lands on a batch boundary (or after `warmUp()` completes) is unaffected, since `_refreshBase()`'s inputs are consistent with a fresh accumulator window in both the original and resumed run at that point.
   - **`samplingRate` is not guaranteed bit-for-bit reproducible across a mid-warm-up resume**, for any subclass, because `_thinningLag()` reads the same never-serialized autocorrelation accumulator. "Bit-for-bit reproducible resume" in this ADR's and issue #1033's sense means the chain's position (`x`) and every subclass's adaptation-derived proposal/step-size/metric parameters reproduce exactly — not `samplingRate`, which is a thinning diagnostic that does not feed back into the chain's trajectory.
5. **Mechanism stays inside the existing constructor + `_internal()`/`initialState.internal` channel** — no static-factory bypass (unlike `Distribution.load()`, ADR-0019) and no new protected hooks. `MCMC` subclass constructors already run subclass-specific validation on `initialState.internal` fields (e.g. HMC's step-size/mass-matrix checks); reusing the same channel means new fields get that validation for free rather than requiring a parallel validation path for a static-factory-style bypass.

## Consequences

**Easier:**
- Resuming a sampler from a saved `state()` after `warmUp()` has completed (the common checkpoint/resume case: crash recovery, distributed computation, deterministic CI) is now bit-for-bit identical to an uninterrupted run, for every subclass.
- `Gibbs` needs no subclass-specific changes at all — it has no `_q` and no adaptation state, so restoring `this.r` in the base class is sufficient.
- Old snapshots (missing `prng`/`prngQ`/accumulator fields) continue to construct valid instances with the pre-#1033 behavior (fresh-seeded PRNG, zeroed accumulators) — additive, non-breaking.

**Harder:**
- `_internal()`'s already-fragile, unenforced naming contract between what it returns and what the constructor reads (ADR-0020's "Harder" consequence, already broken once per `solutions/correctness/2026-07-11-1230-mcmc-state-key-mismatch-silent-sigma-loss.md`) now spans 5-10 fields per subclass instead of 1-3, linearly increasing the contract's surface area with no structural enforcement.
- Every restored field must be validated on the resume channel with the same rigor as the construction channel (per `solutions/correctness/2026-07-15-1230-hmc-resumed-internal-state-validation-gap.md`'s prior bug, where only the `config` channel was validated and `Infinity` reached a loop bound via `initialState.internal`).
- **RWM's mid-batch-window resume is a known, documented gap.** A caller who checkpoints RWM strictly mid-batch and needs its step-size adaptation to be bit-for-bit reproducible from that exact point must wait for a future issue that decouples `_refreshBase()` from the base-class Welford accumulator (or restores that accumulator, which would need its own ADR superseding ADR-0023).
- `samplingRate` divergence after a mid-warm-up resume means two resumed runs' `sample()` calls may thin at different intervals even though their underlying draws are identical — callers relying on `samplingRate` itself being reproducible (as opposed to the draws it thins) are not served by this ADR.
- Test coverage for the new fields must be sized correctly: comparing draws over too short a window after resume can pass even with completely broken accumulator restoration if the window doesn't span a subclass's adaptation batch size (100 for RWM/MALA/Slice) or HMC's dense-metric refresh interval (1000).
