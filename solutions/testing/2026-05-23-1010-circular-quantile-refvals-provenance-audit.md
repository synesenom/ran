# Solution: Circular quantileVals provenance audit and replacement

**Date**: 2026-05-23T10:10:00Z
**Category**: testing
**Related Issue**: #341

## Problem

Twenty `quantileVals` blocks in `test/dist-cases-continuous.js` carried provenance comments of the form `// ranjs CJS bundle; ...`, indicating their reference `{p, x}` pairs were generated by running the ranjs bundle against itself. The `checkQuantileVals` helper (tolerance `|actual-x|/max(1,|x|) < 1e-6`) was asserting "ranjs output matches ranjs output" — a wrong-but-stable `_q` implementation would pass indefinitely with no observable failure.

## Root Cause

When a quantile function has no trivial closed form, the path of least resistance is to run the existing implementation and record its output. Because the values were in fact numerically correct in this instance (max relative error vs. independent sources < 4e-10), the circular provenance was invisible until explicitly audited. The test encoded current behavior rather than verified-correct behavior.

## Fix

All 20 blocks were replaced using a three-tier hierarchy of independent sources:

1. **Closed-form exact** — the distribution's `_q` algebraic formula evaluated in Python/mpmath dps=50:
   - ExponentialLogarithmic, GammaGompertz, LogisticExponential, ShiftedLogLogistic, UQuadratic
   - Exact rational/integer values were recovered where the formula yields them (ShiftedLogLogistic Q(0.75)=8.0, Q(0.95)=360.0, Q(0.99)=9800.0)

2. **Known-distribution transform in scipy** — target distribution is a measurable function of a scipy-native distribution:
   - LogGamma: `exp(Gamma(alpha, scale=1/beta).ppf(p)) + mu - 1`
   - StudentZ(n): `t(df=n-1).ppf(p) / sqrt(n-1)`

3. **scipy or mpmath CDF inversion** — closed-form CDF inverted numerically by an independent tool:
   - mpmath `mp.findroot` at dps=50: BenktanderII, GeneralizedExponential, Lindley, Makeham, Muth
   - `scipy.optimize.brentq` on closed-form CDF: BetaRectangular, Logarithmic, RaisedCosine, Slash
   - Poisson-mixture of scipy-native CDFs: DoublyNoncentralBeta, DoublyNoncentralF, DoublyNoncentralT, NoncentralBeta

Each block's provenance comment was updated to name the independent source and formula used, for example:
```js
// mpmath dps=50, CDF inversion: F(x)=1-x^(b-1)*exp(a*(1-x^b)/b) (a=2, b=0.9995)
// closed-form: Q(p) = log(1 + (p/(1-p))^(1/kappa)) / lambda (lambda=2, kappa=2)
// exp(scipy.stats.gamma(a=2, scale=0.5).ppf(p)) + mu - 1; Y=exp(G)+mu-1, G~Gamma(2, rate=2)
```

A triage check during review caught a parenthesization error in the BenktanderII comment draft. The initial formula `F(x)=(1-x^(b-1))*exp(...)` was wrong — the `1-` belongs outside the product: `F(x)=1-x^(b-1)*exp(a*(1-x^b)/b)`.

## Prevention Strategy

Enforce a source provenance rule at test-case authoring time: every `quantileVals` block must carry a comment that names a source outside the ranjs codebase. Acceptable forms:
```js
// scipy.stats.X(params).ppf(p)
// closed-form: Q(p) = <formula>
// mpmath dps=50, CDF inversion: F(x) = <formula>
// <transform>: X = f(Y), Y ~ <known distribution>
```

A CI grep check enforces this mechanically:
```bash
grep "ranjs CJS bundle" test/dist-cases-continuous.js && exit 1 || exit 0
```

When adding a new distribution with a non-trivial `_q`, apply the three-tier hierarchy: prefer closed-form → known-distribution transform → mpmath/scipy CDF inversion. Never use the library's own output as the reference.

## Key Insight

A `quantileVals` block whose reference values were produced by the library under test is a non-test — it enforces stability, not correctness — and must be replaced by values traceable to scipy, mpmath at high precision, or a documented closed-form formula before it can detect a regression.
