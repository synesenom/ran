# Solution: refVals Self-Validation at Cancellation Boundaries

**Date**: 2026-05-22T17:08:00Z
**Category**: testing
**Related Issue**: #295

## Problem

Two `refVals` rows for `BenktanderII` at `x = 1+1e-6` and `x = 1+1e-4` (params `[2, 0.9995]`) carried 16–19 significant digits and a comment citing "Python Decimal @ 60 dps". However, those values had been generated by evaluating the JS `expm1`-based implementation formula in float64 (or by a Python Decimal script that replicated the same `expm1` path), not by an independent direct-formula computation. The test was validating the implementation against its own output — a tautology dressed as a reference check.

A subtle regression in the cancellation-avoidance path (`(1 - xbm1) - xbm1 * expm1(u)`) would produce wrong CDF values near `x=1`, yet pass the `checkRefVals` harness undetected because the expected values encoded the same wrong computation.

## Root Cause

When a cancellation fix is added to `_cdf` (e.g. replacing `1 - x^(b-1)*exp(u)` with the numerically stable split `(1-xbm1) - xbm1*expm1(u)`), the near-boundary rows in `refVals` are naturally generated by running the *fixed* implementation. The values look high-precision because they have many decimal digits, but those digits trace back to the same code path being tested. Any error in the `expm1` split cancels between expected and actual, so the test provides zero protection against regressions in that exact path.

The problem is structurally invisible: a reader sees a big-decimal value with 18 digits and assumes it was independently validated. The comment may even name a tool ("Python Decimal") without specifying whether the tool ran the direct formula or the same cancellation-mitigated expression.

## Fix

The two near-boundary `refVals` rows were recomputed via Python `Decimal` at 60 decimal places using the **direct mathematical formula**:

```
F(x) = 1 - x^(b-1) * exp(a*(1-x^b)/b)
```

This formula uses no `expm1` and is arithmetically independent of the JS `_cdf` implementation path. Each row now carries an explicit inline comment with the exact Python expression used:

```js
// Python Decimal @ 60 dps: xb=Decimal('1.000001')**b; u=a*(1-xb)/b; 1-(xb/x)*u.exp()
{ x: 1.000001, pdf: 2.0004959965037585, cdf: 2.0004979982512112e-06 },
```

The comment names the formula, not just the tool, making provenance verifiable without running the script.

## Prevention Strategy

**Rule**: for any `_cdf` or `_pdf` implementation that contains a cancellation fix (expm1, log1p, split-sum), `refVals` rows in the cancellation-active region must be sourced from a formula that does **not** use the same fix — typically Python `Decimal`/`mpmath` using the direct mathematical definition.

Checklist when adding `refVals` near a known cancellation boundary:
1. Identify which formula the JS code uses (e.g. `expm1`-based split).
2. Write a Python script using the *direct* formula (no `expm1`, no `log1p`), evaluated at ≥50 decimal places.
3. Add an inline comment on each row with the exact Python expression, not just `// mpmath` or `// ranjs CJS bundle`.
4. Red-flag any near-boundary row tagged `// ranjs CJS bundle` or sourced without an explicit formula — regenerate it.

The numerical values of correctly rounded float64 results may be identical (the bug would only appear if the cancellation fix had a coefficient error), which is exactly why provenance must be documented: the independence is the value, not the digits themselves.
