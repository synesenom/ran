# ADR-0024: MCMC Warm-Up Stays Fixed-Length; Convergence Is a Post-Hoc, Multi-Chain Concern

**Date**: 2026-07-13
**Status**: Accepted

## Context

`MCMC.warmUp(progress, maxBatches = 100)` (`src/mc/_mcmc.js:173-183`) runs a fixed schedule — `maxBatches` batches of 10,000 iterations each (1,000,000 draws at the default) — with no stopping condition tied to any statistic computed during the loop. Issue #921 asked whether this is the right design, naming two risks: wasted computation when adaptation converges fast, and a silent under-tuned sampler with no warning when it converges slowly.

Two signals are already computed during `warmUp()` that a stopping heuristic could in principle read:

- **`RWM._ls`**, the global log proposal-scale, adapted via batch Robbins-Monro toward a target acceptance rate (`src/mc/rwm.js:97-114`, ADR-0022). This is subclass-private — the base `MCMC` class has no generic notion of "the adapted proposal scale," and making it visible to a base-class heuristic would require a fourth subclass-contract hook beyond `_iter`/`_adjust`/`_internal` (ADR-0020 §2), which that ADR explicitly states has not been needed for any sampler family scoped so far (`RWM` shipped; Gibbs/HMC/NUTS scoped in issues #821/#824-826).
- **`samplingRate`/`_thinningLag()`**, the base-class thinning-lag estimate, is generic and already recomputed once per batch (`src/mc/_mcmc.js:263-288`). But ADR-0020 §3 already documents this exact signal as noisy and non-stationary during warm-up: the underlying autocorrelation accumulators are never reset between batches, so the reported lag "can shift meaningfully from one batch to the next as adaptation proceeds," which is why `_adjustSamplingRate()` damps changes to ±1 per batch instead of jumping to the estimate directly.

Both signals describe whether *the proposal has stopped adapting*, not whether *the chain has explored the target distribution*. The MCMC literature is explicit that these are different questions: Gelman & Rubin (1992) established that a single chain can appear internally stable — step size plateaued, acceptance rate steady, thinning lag unchanging — while still being trapped near its initial state or in one mode of a multimodal target, because a lone chain has no independently-initialized reference to compare against. A stuck chain and a converged chain can be statistically indistinguishable from the inside. This is precisely why Gelman-Rubin's R-hat statistic requires ≥2 independently-initialized chains: R-hat is large exactly when chains disagree, and only meaningfully close to 1 when independent chains have mixed into agreement — a signal no single chain can produce on its own.

External precedent confirms this is not merely a theoretical concern that a good implementation can engineer around. Stan's HMC/NUTS warm-up (mc-stan.org, Reference Manual §15.2, "HMC Algorithm Parameters") uses a fixed, bounded three-phase window schedule — an initial fast interval (default 75 iterations) for step-size-only adaptation, a series of expanding memoryless slow windows (default initial width 25, doubling) for metric/(co)variance estimation, and a final fast interval (default 50) to re-tune step size to the final metric. This schedule is sized by iteration counts, not triggered by any in-loop convergence check. Stan computes R-hat only after warm-up, across independently-run chains, as a post-hoc pass/fail gate — never as a single-chain stopping rule. Every mainstream MCMC toolkit surveyed (Stan, PyMC — see research doc "Sources") follows this same split: bounded schedule for adaptation, multi-chain R-hat for convergence.

`ranjs` already has the pieces for the correct pattern, just not wired together: `gelmanRubin(samples, maxLength)` (`src/mc/gelman-rubin.js`) is a complete, standalone R-hat implementation, but it is never called from `_mcmc.js`/`rwm.js` — it takes plain `number[][]` chain arrays and is exercised in tests only by hand-constructing multiple independently-seeded `RWM` instances, running `warmUp()`/`sample()` on each, and passing the results in (`test/mc.js:474-561`). `MCMC` has no multi-chain concept at all: every instance owns exactly one PRNG and one state vector.

Three design options were evaluated via a design-propose/design-critique pass (full detail in `thoughts/research/2026-07-13-1714-mcmc-warmup-convergence-signal.md` and the corresponding plan document):

1. **Keep `warmUp()` fixed-length; add a standalone multi-chain runner helper** (`runChains()`, pattern-matching `gelmanRubin()`) that constructs N independently-seeded chains, warms up and samples each, and calls `gelmanRubin()` on the result — making the literature-correct workflow easy to use without adding multi-chain logic to the single-chain `MCMC` base class.
2. **Add an in-loop early-stopping heuristic** to `warmUp()` itself, exiting early once `samplingRate`/`_thinningLag()` stabilizes for a configurable number of consecutive batches.
3. **Hybrid**: option 1's helper, plus an enriched `warmUp()` progress callback that can return `false` to request early exit, framed as an advanced escape hatch rather than a convergence guarantee.

Both the proposing and critiquing passes independently converged on option 1 with high confidence. Option 2 was rejected outright: it would build a stopping criterion on top of a signal ADR-0020 itself already documents as unreliable during the exact phase it would be used in, and it fails on precisely the failure mode the issue describes — a chain stuck in a local mode shows a stable acceptance rate and a stable thinning lag, so the heuristic would report "converged" on exactly the inputs where it is most wrong. This is not a corner case to be mitigated with a larger patience window; it is the defining failure mode single-chain heuristics cannot detect by construction. Option 3's escape hatch was found to duplicate functionality the codebase already exposes: `iterate()`'s existing JSDoc (`src/mc/_mcmc.js:143-148`) already documents "custom stopping rules" as the intended use case for callers who want to drive their own loop, and no other callback in the codebase (`sample()`'s progress callback included) uses a return value for flow control — introducing that protocol for a single method would be a novel, asymmetric convention paid for by real backward-compatibility test surface (distinguishing legacy number-returning callbacks from new object-returning ones) and TypeScript-declaration overload complexity (ADR-0010), for a capability that adds no statistical rigor over option 2's already-rejected heuristic.

## Decision

`MCMC.warmUp()` remains a fixed-length, non-convergence-gated schedule. No early-stopping heuristic, in-loop or callback-triggered, will be added to it. This is a deliberate acknowledgment that no signal computable from a single chain — adapted proposal scale, thinning lag, acceptance rate, or any combination of them — can distinguish "converged" from "stuck," and that shipping a heuristic which cannot make this distinction would be actively misleading: a stuck chain is exactly the input on which such a heuristic reports false confidence.

Convergence assessment is a **multi-chain, post-hoc concern**, kept out of the single-chain `MCMC` base class per ADR-0020 §2's established principle of not adding cross-cutting machinery until a concrete need forces it. The recommended workflow is: construct multiple independently-seeded samplers, run `warmUp()`/`sample()` on each, and gate on `gelmanRubin()` across the resulting chains. A follow-up implementation issue will add a standalone `runChains()`-style helper (mirroring `gelmanRubin()`'s existing standalone-function pattern in `src/mc/`) that performs this construct/warm-up/sample/R-hat sequence for the caller, so the correct pattern does not have to be hand-rolled on every use — but introduces no change to `MCMC`, `RWM`, or `gelmanRubin()` themselves, and no new subclass-contract hook.

This issue (#921) is research/design-only; no code changes ship with this ADR. The follow-up implementation issue is scoped to adding the `runChains()` helper and its tests only.

## Consequences

**Easier:**
- The single-chain pitfall is now documented at the point future contributors are most likely to be tempted to "fix" `warmUp()`'s fixed budget — this ADR, linked from `_mcmc.js`, heads off a class of well-intentioned but statistically unsound PRs (e.g., "stop warm-up once the acceptance rate stabilizes") before they're written.
- `MCMC`/`RWM` stay unchanged: no new constructor options, no new subclass hook, no callback-protocol change. ADR-0020's subclass contract (`_iter`/`_adjust`/`_internal`) remains sufficient for every sampler family scoped so far and for the upcoming `runChains()` helper, which is a pure consumer of the existing public API (`seed()`, `warmUp()`, `sample()`, `gelmanRubin()`).
- The upcoming `runChains()` helper has a direct precedent to follow (`gelman-rubin.js`'s standalone-function shape), keeping its design and review small and low-risk.

**Harder:**
- Callers who want *any* automatic signal for "has warm-up run long enough" still have to either run multiple chains (more compute, `runChains()` once implemented makes this convenient but not free) or fall back to manually calling `iterate()` in their own loop with their own — necessarily single-chain, necessarily heuristic — stopping rule, fully aware of this ADR's stated limitation.
- `maxBatches` remains a caller-supplied constant with no relationship to any measured statistic; a caller with no prior intuition for how many batches a given target needs must still guess, run `runChains()`, and check R-hat after the fact rather than getting an in-loop answer.
- If a future sampler family (Gibbs, HMC, NUTS — issues #821/#824-826) turns out to need a materially different warm-up schedule shape (e.g., Stan's three-phase windowed schedule rather than ranjs's flat batch loop), that is a separate, larger design question this ADR does not resolve — it only settles that *convergence-gated* early stopping is rejected, not that the flat-batch *schedule shape* itself is final for every future family.
